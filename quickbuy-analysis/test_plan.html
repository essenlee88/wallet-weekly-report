1. 背景与目标

用户反馈在高竞争场景（New Token / 毕业抢开盘）下 Quickbuy「抢不过别人」。本方案旨在：

  1. 量化全链路每一段的延迟——定位瓶颈在哪里
  2. 衡量不同网络路径的影响——VPN 出口、弱网、多跳对成功率的实际影响
  3. 与竞品做受控对比——同一 Token、同一时刻、多平台同测
  4. 输出数据驱动的优化建议——给交易侧明确的改进方向和优先级

交易成功的定义

结果定义判定依据
成功 交易上链且 swap 实际执行，用户获得目标 Token 链上 tx status = success 且包含 swap event
失败 — 链上 revert 交易上链但执行失败（滑点超限、流动性不足等） 链上 tx status = failed，记录 revert 原因
失败 — 未上链 交易未被打包上链（blockhash 过期、RPC 丢弃、Priority Fee 过低被挤出） 提交后超时无 tx hash，或 tx hash 查无记录
失败 — 客户端错误 交易未发出（网络超时、API 报错、签名失败） 客户端返回错误提示，无 tx hash

所有场景（A1-A4）均统计交易成功率 = 成功次数 / 总尝试次数。失败时必须记录失败类型和具体原因。

关键发现（调研阶段）：Bonding Curve 打满后、迁移交易执行前（即 BC 满额待迁移阶段），BC 合约仍未锁定。GMGN / Binance 在此阶段仍走 BC 直接买入（立即成交），Axiom 则跳过 BC 直接排队 DEX Snipe（等池子上线）。两种路径的成交价和风险不同，需要在测试中验证哪种路由策略更优。

2. 全链路模型

一笔 Quickbuy 交易从 Token 诞生到用户收到成交确认，经历以下 6 段延迟：

T0 Token 上链（block timestamp） │ △1 发现延迟 ← 链上事件监听 + 数据管道处理 T1 我方数据管道检测到新 Token │ △2 推送延迟 ← WebSocket / API 推送到客户端 T2 客户端展示新 Token（用户可见） │ △3 用户网络延迟 ← 用户设备 → [VPN 出口] → 平台 API 服务器 T3 Buy 请求到达 API 服务器 │ △4 服务端处理 ← 构造 tx + 签名（MPC）+ 发送到 RPC T4 Transaction 发送到 RPC 节点 │ △5 链上执行 ← RPC → Leader Validator → 打包上链 T5 Transaction 上链确认（on-chain timestamp） │ △6 确认回传 ← 链上确认 → 服务端 → 客户端展示 T6 用户收到成交结果
段延迟段影响因素可控性备注
△1发现延迟链上事件订阅方式、数据管道架构我方可优化决定「谁先看到新币」
△2推送延迟WebSocket 连接质量、服务端推送逻辑我方可优化—
△3用户网络延迟用户地理位置、VPN 出口、VPN 跳数、网络质量用户侧，我方无法控制本次重点测试项
△4服务端处理tx 构造逻辑、MPC 签名延迟、服务器位置我方可优化含 API 服务器 → RPC 节点的内网延迟
△5链上执行RPC 节点位置和质量、Priority Fee、网络拥堵部分可优化（RPC 选择、Fee 策略）Jito Bundle 可提升优先级
△6确认回传确认订阅方式我方可优化影响用户体验但不影响成交

4. 网络路径测试

维度 C — VPN 出口节点

我们的核心用户群在中国大陆，必须经 VPN 访问。不同 VPN 出口到平台 API 服务器的延迟差异显著。

编号VPN 出口到美东 RTT（估）到亚太 RTT（估）说明
C1 🇯🇵 日本（东京） ~160ms ~30ms 亚太用户最常用出口；到亚太服务器快，到美东较慢
C2 🇺🇸 美国（US East / West） ~10-30ms ~180ms 如果平台 API / RPC 在美东，这是最优出口
C3 🇩🇪 法兰克福 ~80ms ~200ms 欧洲节点；Solana validator 密集区之一
C4 🇭🇰 香港 ~190ms ~10ms 物理距离最近但 GFW 干扰风险
C5 🇸🇬 新加坡 ~230ms ~20ms 亚太常用出口；Binance 基础设施重要节点
C0 （基准）无 VPN — 美东直连 — — 对照组，排除 VPN 开销

网络路径 × 延迟链路分析

以大陆用户 → 日本 VPN → 平台 API（假设美东）为例： 用户设备（大陆） │ ~50-80ms ← 大陆 → 日本 VPN 节点（单跳） │ ~30-50ms ← VPN 加密/解密开销 日本 VPN 出口 │ ~150ms ← 日本 → 美东（跨太平洋） 平台 API 服务器（美东） │ ~1-5ms ← 同机房 / 同区域 RPC 节点 │ ~10-50ms ← RPC → 当前 Leader Validator Solana Leader ──────────────────────────────── △3 总计 ≈ 230-330ms（用户点击到请求到达 API） 对比：美东直连用户 △3 ≈ 10-30ms → 差距 200-300ms 对比：多跳 VPN（大陆→日本→美国）△3 ≈ 350-500ms

测试方法

在各 VPN 出口节点上部署测试环境（或使用同一台设备切换 VPN 节点），对同一 Token 执行 Quickbuy，记录各段耗时。

测试项方法指标
各出口到平台 API 的 RTT 每个出口 ping 平台 API 域名 100 次 P50 / P90 / P99 RTT + 丢包率
各出口的 Quickbuy 端到端延迟 在 A4 场景下，每个出口执行 ≥10 次买入 T_total 的 P50 / P90 + 成功率
高竞争场景下各出口的差异 在 A1/A3 场景下，2-3 个出口同时买入同一 Token 成交排序差异 + 价差

6.1 多平台同步实测（主方法）

环境准备

项目要求

设备6 个浏览器窗口（或设备），同一网络环境
平台Binance Web3 / GMGN / Axiom / Terminal.gg / OKX Web3 /（预留 1 个补测位）

预设统一组：相同买入金额 + 滑点 + Fee；默认组：各平台默认配置

统一参数

参数统一组默认组
买入金额0.01 SOL / 0.01 BNB / 0.001 ETH同

Priority Fee手动统一 0.001 SOL各平台默认

执行流程（每轮）

  1. 所有窗口同屏排列，录屏开始
  2. 监控新 Token 出现（以链上事件或最快平台的 feed 为准）
  3. 新 Token 出现后 → 所有窗口尽可能同时点击 Quickbuy
  4. 记录：发现时间、点击时间、确认/失败时间、失败原因
  5. 获取所有交易的 tx hash，链上核查 block 位置和实际成交价
  6. 填入记录表

6.2 网络路径测试

  1. 基准测试（C0-C5）：同一台设备，切换 VPN 出口节点，在 A4（Migrated 稳态）场景下各执行 10+ 次 Quickbuy
  2. 弱网测试（W1-W2）：在最优 VPN 出口上叠加弱网模拟，在 A4 场景下各执行 10+ 次 Quickbuy
  3. 高竞争 × 网络路径：选 2-3 个代表性出口（如日本 + 美国 + 新加坡），在 A1/A3 场景下同时买入同一 Token

6.3 链上数据回溯（辅助验证）

  1. 实测交易链上深挖：Block 内排序位置、是否被 sandwich、实际 Gas 消耗
  2. 同 Token 其他交易者分析：前 20 笔交易中各平台占比和排序
  3. 批量统计：近 7 天新 Token 前 5 笔交易的平台分布（大样本验证实测结论）
