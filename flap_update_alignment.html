核心变化

零税 token 毕业后升级为 PCS Infinity，获得集中流动性 + Hook 管理能力

原有零税路径（TOKEN_V2_PERMIT）毕业后落入 PancakeSwap V2 静态 LP。新版本毕业后进入 PCS Infinity Vault 的 CL 架构，并由 Default Flap Hook 在 swap / addLiquidity 等生命周期节点主动介入，提供更好的深度管理。

安全评估 — PCS Infinity Hook 架构
高
Default Hook 单点故障 & 升级风险
所有零税 token 共享同一个 Default Flap Hook（0xF1A9...c1c1）。Hook 可修改手续费、限制操作、产生 hookDelta（交易中抽取费用，同 tx 内结算）。若 FLAP 团队恶意升级或 Hook 合约存在漏洞，将波及所有已毕业零税 token 的交易。
当前 Hook 是否可升级、升级权限归属，尚未在公开文档中说明——需向 FLAP 确认。
中
Hook 参数被协议方随意修改
FLAP 理论上可通过升级 Hook 开启高额手续费或限制卖出，损害用户公平性。
低
项目方（dev）做恶空间
当前 Default Hook 由 FLAP 协议统一管理，项目方无法替换或自定义 Hook，对已毕业 token 的合约行为无直接控制权。做恶空间来自 FLAP 协议层，不来自个别项目方。
SDL 提交要点： Hook 合约地址 0xF1A9...c1c1 的可升级性 + 权限归属；hookDelta 抽费机制；用户交易路径中是否存在不可预期的资金扣留风险。
Default Hook — Dev 自定义空间

PCS Infinity Hook 架构本身支持：动态手续费、限制 LP 操作、限制 swap 方向/金额、hookDelta 抽费。但 FLAP Default Hook 实际开放给项目方的自定义空间，目前文档未说明。

待向 FLAP/Cedric 确认：

  • Default Hook 是否对项目方暴露任何可配置参数（如费率上限、流动性区间偏好）？
  • 项目方是否可通过白名单机制替换为自定义 Hook？
  • Default Hook 合约是否可升级（proxy 模式）？升级权在谁手里？
