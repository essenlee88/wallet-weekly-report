FLAP 零税 Token 更新说明

TOKEN_V3_PERMIT · PCS Infinity 流动性升级 · 2026-05-15
核心思想

此次更新专门针对零税 token，毕业后导向 PCS Infinity，获得更好的流动性管理能力

FLAP 原有的零税路径（TOKEN_V2_PERMIT）毕业后落入普通 PancakeSwap V2 池，流动性是被动静态的。新版本（TOKEN_V3_PERMIT）毕业后进入 PCS Infinity Vault 的集中流动性架构，LP 可以将资金集中在实际交易价格区间，同等资金深度下成交体验更好，并通过 Default Flap Hook 在关键节点主动干预流动性行为。

Default Flap Hook — 流动性管理机制（调研结果）

Hook 实际做了什么？

来源：FLAP 官方文档 docs.flap.sh（IPortal.sol / IV4Locker 接口）+ PancakeSwap infinity-core ICLHooks.sol · 2026-05-16

一句话能懂版

旧版（V2 静态 LP）：资金平铺在"零到无穷大"的所有价格区间，大部分钱放在没人交易的地方，实际深度差。

V3 对比 V2 的核心升级只有一点：

  1. 钱集中放在有用的地方 — 把 LP 拆成两堆：一堆只放报价币（BNB/USDT）放在当前价格左侧，一堆只放项目 token 放在右侧。同样的钱，集中在真实交易区间，滑点更小。（V2 静态 LP 是全区间平铺，这是 V3 唯一核心升级。）
手续费再投入不是 V3 的优势：V2 手续费本来就自动留在池子里叠加（机制内置）。V4 CL 架构把手续费单独记账（Uniswap V3/V4 通用特性），导致手续费不再自动回池，需要主动收割。FLAP 用 Hook + permissionless keeper 补回了这个行为——是在 V4 架构下维持与 V2 的平价，不是新增优势。
锁仓也不是 V3 新增：FLAP V2 也有 GoPlus / UNCX 锁仓，协议层通用机制。

已确认：

机制 说明
双 position 拆分结构 LP 分成两个独立头寸：lower（quote-only，仅含报价币） + upper（token-only，仅含项目 token）。模拟 V3 单边流动性，将资金集中在有效价格区间两侧，避免全范围低效分布。
流动性永久锁仓 LP 通过 GoPlus 或 UNCX locker 锁定，Portal 作为 lock owner。Dev 无法撤出流动性。
⚠ V2 也有此机制，非 V3 新增，是 FLAP 协议层通用能力。
手续费收集与分发 collectV4Fees(token) 任何人均可调用（permissionless keeper）。费用路径：GoPlus/UNCX Locker → Portal → TaxProcessor → 分发给各方。
来源：docs.flap.sh IPortal.sol / IV4Locker 接口注释
流动性追加（再投资） addV4LPLiquidity(token, tokenAmount, quoteAmount) 由 TaxProcessor 调用，将 LP-share 税收收益再投入两个 position，深度随时间增长。
来源：docs.flap.sh IPortal.sol / IV4Locker 接口注释

Default Flap Hook — Dev 自定义空间

Hook 给了项目方（dev）哪些空间？

PCS Infinity Hook 架构允许在 14 个流动性生命周期节点（swap 前后、addLiquidity 前后、removeLiquidity 前后等）注入自定义逻辑。FLAP 部署了一个统一的 Default Flap Hook（0xF1A9...c1c1），所有零税 token 共享此 Hook。
